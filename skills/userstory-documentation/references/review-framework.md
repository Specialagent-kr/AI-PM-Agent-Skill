# 유저스토리 검증 프레임워크 상세

## Part 1: 3 C's 상세 기준

### 1. Card — 스토리가 잘 작성되었는가?

**체크리스트:**
- [ ] "As a [사용자], I want [목표], so that [혜택]" 형식 준수
- [ ] 제목만으로 내용 파악 가능
- [ ] 설명이 간결하고 명확

**질문:**
- 표준 유저스토리 템플릿을 따르는가?
- 제목만으로 스토리를 이해할 수 있는가?
- 설명이 짧지만 포괄적인가?

---

### 2. Conversation — 충분한 논의가 이루어졌는가?

**체크리스트:**
- [ ] PO, 개발자, QA, 디자이너와 스토리 논의 완료
- [ ] 논의 노트, 질문, 결정 사항 문서화
- [ ] 모든 이해관계자가 의견 제시

**질문:**
- 팀 논의의 증거가 있는가?
- 대화 노트가 스토리에 기록되어 있는가?
- 모든 질문이 해소되었는가?

---

### 3. Confirmation — 수락 기준이 명확한가?

**체크리스트:**
- [ ] AC가 "완료" 시점을 명확히 정의
- [ ] AC가 측정 가능하고 테스트 가능
- [ ] Happy path와 오류 케이스 모두 포함

**질문:**
- QA가 이 AC로 테스트 케이스를 작성할 수 있는가?
- AC가 "빠르다" 대신 "2초 이내 로드"처럼 구체적인가?
- 정상 플로우와 엣지 케이스를 모두 다루는가?

**예시:**

❌ **나쁜 AC:**
- "사용자가 카드를 저장할 수 있다"
- "시스템이 안전하다"

✅ **좋은 AC:**
- "사용자가 유효성 검사와 함께 16자리 카드 번호 입력 가능"
- "전송 중 TLS 1.3 암호화로 카드 데이터 보호"
- "카드 번호 잘못 입력 시 오류 메시지 표시"

## Part 2: INVEST 원칙 상세

### I — Independent (독립적)

**Pass 기준:** 다른 스토리 없이 개발·테스트·배포 가능

**질문:**
- 이 스토리는 다른 스토리 없이 작업 가능한가?
- 의존성이 있다면 스토리를 통합하거나 재구성해야 하는가?

**지침:** "Part 1", "Part 2" 스토리 지양. 각 스토리는 독자적 가치 전달.

---

### N — Negotiable (협상 가능)

**Pass 기준:** 결과(What)에 집중하고 구현(How)은 팀이 논의 가능

**질문:**
- 스토리가 구현을 규정하는가, 아니면 원하는 결과를 설명하는가?
- 기술적 논의와 대안의 여지가 있는가?

❌ **너무 규정적:** "PostgreSQL의 cards 테이블에 사용자 카드 저장"
✅ **협상 가능:** "사용자가 향후 사용을 위해 결제 수단 저장 가능" (팀이 저장 방식 결정)

---

### V — Valuable (가치 있음)

**Pass 기준:** 최종 사용자 또는 비즈니스에 명확한 가치 전달

**질문:**
- 이것이 사용자에게 어떤 가치를 주는가?
- 기술적 스토리라면 어떻게 사용자 가치를 가능하게 하는가?
- 비기술 이해관계자에게 혜택을 설명할 수 있는가?

❌ **가치 불명확:** "결제 처리 모듈을 새 아키텍처로 리팩토링"
✅ **가치 명확:** "사용자가 결제 수단을 저장해 반복 구매 시 빠른 결제" (기술 리팩토링은 이 스토리의 Task)

---

### E — Estimable (추정 가능)

**Pass 기준:** 개발팀이 작업량을 추정 가능한 충분한 정보 있음

**질문:**
- 개발팀이 스토리 포인트/작업량을 추정할 수 있는가?
- 추정을 막는 주요 미지수가 있는가?
- 추정 불가 시 스파이크(조사 스토리)가 필요한가?

**지침:** 추정 불가 시 내용 명확화 또는 리서치가 필요. 조사를 위한 별도 스파이크 스토리 생성.

---

### S — Small (작은 크기)

**Pass 기준:** 1 스프린트(1-2주) 내 완료 가능

**질문:**
- 1 스프린트 내 완료 가능한가?
- 너무 크다면 더 작은 수직 슬라이스로 분해 가능한가?
- 너무 작다면 관련 스토리와 결합해야 하는가?

**예시 — 큰 스토리 분해:**

🔴 **너무 큰 규모 (Epic):**
"사용자가 모든 결제 수단을 관리"

✅ **적절한 크기:**
1. "사용자가 저장된 결제 수단 목록 조회"
2. "사용자가 새 결제 수단 추가"
3. "사용자가 저장된 결제 수단 삭제"
4. "사용자가 기본 결제 수단 설정"

---

### T — Testable (테스트 가능)

**Pass 기준:** 완료 여부를 객관적으로 검증 가능한 기준 존재

**질문:**
- QA가 이 스토리의 완료를 객관적으로 테스트할 수 있는가?
- AC가 테스트 케이스 작성에 충분히 구체적인가?
- "완료"의 의미에 모호함이 없는가?

❌ **테스트 불가:** "결제가 사용자 친화적이어야 한다"
✅ **테스트 가능:** "사용자가 3단계 이하로 결제 완료 / 4G 연결에서 결제 페이지 2초 이내 로드"

## Part 3: 추가 점검 상세

### 사용자 페르소나
- "user" 또는 "customer" 같은 일반적 표현 금지
- 예: "로그인한 회원", "게스트 방문자", "관리자"

### UI/UX 디자인
- Figma, Sketch 등 와이어프레임/목업 첨부
- 디자인 보류 중이면 미해결 질문 명시
- 시각적 요구사항 명확화

### 비기능 요구사항
- 성능 요구사항 (해당 시)
- 보안 고려사항
- 데이터 프라이버시 및 컴플라이언스
- 접근성 요구사항

**예시:**
- "4G 연결에서 2초 이내 페이지 로드"
- "저장 및 전송 중 카드 데이터 암호화"
- "WCAG 2.1 AA 기준 충족"

### Definition of Ready (DoR)
팀의 DoR 체크리스트 충족 여부 확인. 일반적 DoR 요소:
- [ ] 유저스토리 형식 준수
- [ ] 수락 기준 정의
- [ ] 의존성 식별
- [ ] 디자인 자산 첨부 (해당 시)
- [ ] 팀이 작업량 추정 완료
- [ ] 주요 블로커 없음
