# 유저스토리 작성 상세 가이드

## 수락 기준(AC) 포맷 옵션

### 옵션 1: Given-When-Then (Gherkin)
```
Given [초기 컨텍스트/전제 조건]
When [이벤트 또는 액션 발생]
Then [기대 결과]
```
**예시:**
```
Given 저장된 결제 수단이 있는 로그인 사용자인 경우
When 결제 페이지로 이동하면
Then 저장된 결제 수단 목록이 옵션으로 표시된다
```

### 옵션 2: 시나리오 기반
```
시나리오: [이름]
- [기준 1]
- [기준 2]
- [기준 3]
```
**예시:**
```
시나리오: 새 신용카드 저장
- 사용자가 카드 번호, 유효기간, CVV 입력 가능
- 시스템이 카드 형식 유효성 검사
- 뒷자리 4자리 표시와 함께 카드 저장
- 사용자에게 확인 메시지 표시
```

### 옵션 3: 체크리스트
```
스토리 완료 조건:
- [ ] [기준 1]
- [ ] [기준 2]
- [ ] [기준 3]
```

---

## AC 작성 가이드라인

- 테스트 가능하고 측정 가능하게 작성
- Happy path(정상 플로우)를 포함
- 엣지 케이스와 오류 시나리오 포함
- 구체적으로: "2초 이내 로드" (O), "빠르게 로드" (X)
- 구현 세부사항 지양
- 사용자 관점에서 작성

**좋은 AC vs 나쁜 AC:**

❌ **나쁜 AC:**
- "시스템이 빠르다"
- "좋은 사용자 경험"
- "올바르게 작동한다"

✅ **좋은 AC:**
- "4G 연결에서 2초 이내 페이지 로드"
- "데이터 로드 중 로딩 인디케이터 표시"
- "네트워크 실패 시 재시도 옵션과 함께 오류 메시지 표시"

## 수직 슬라이싱 예시

❌ **수평 슬라이싱 (잘못된 방법):**
- Story 1: 데이터베이스 스키마 생성
- Story 2: API 엔드포인트 생성
- Story 3: 프론트엔드 UI 구축

✅ **수직 슬라이싱 (올바른 방법):**
- Story 1: 사용자가 저장된 카드 목록을 볼 수 있다 (단순 목록)
- Story 2: 사용자가 새 카드를 추가할 수 있다 (유효성 검사 포함)
- Story 3: 사용자가 저장된 카드를 삭제할 수 있다 (확인 포함)

## 완성 예시: 결제 수단 저장

```
제목: 향후 사용을 위한 결제 수단 저장

As a 로그인한 고객
I want to 내 신용카드 정보를 안전하게 저장하기
So that 향후 구매 시 더 빠르게 결제할 수 있다

컨텍스트:
자주 쇼핑하는 고객은 결제 정보를 반복 입력하는 것을 원하지 않는다.
이 기능을 통해 안전하게 결제 수단을 저장하고 원클릭 결제를 지원한다.

수락 기준:
Given 결제 페이지의 로그인 사용자인 경우
When 카드 정보 입력 후 "나중에도 사용" 체크 시
Then 카드가 암호화되어 저장되고 뒷자리 4자리가 표시된다

Given 저장된 결제 수단이 있는 경우
When 결제 페이지로 돌아오면
Then 저장된 카드들이 결제 옵션으로 표시된다

Given 저장된 카드에서 "삭제" 선택 시
When 삭제를 확인하면
Then 카드가 삭제되고 더 이상 표시되지 않는다

메모:
- Figma: [결제 UI 목업 링크]
- 보안: PCI-DSS 준수 암호화 필수
- 의존성: 결제 게이트웨이 연동 (Story #123)
```

## 흔한 실수

| 실수 | 잘못된 예 | 올바른 예 |
|------|----------|----------|
| 너무 기술적 | "REST API 엔드포인트 구현" | "사용자가 주문 내역을 볼 수 있다" |
| 너무 큰 규모 | "사용자가 전체 계정을 관리" | 프로필/설정/결제로 분리 |
| 구현 중심 | "React hooks와 Redux 사용" | "사용자가 실시간 주문 상태를 본다" |
| 모호한 AC | "시스템이 빠르다" | "4G 연결에서 2초 이내 로드" |
| 가치 없음 | "코드 리팩토링" | "사용자가 날짜 범위로 주문 검색" |
