# Jobs-to-be-Done (JTBD)

## 개요

JTBD는 사용자가 제품을 사용하는 진짜 이유를 이해하는 프레임워크다.
사람들은 제품을 "구매"하는 것이 아니라 특정 "일(Job)"을 완수하기 위해 제품을 "고용"한다.

**핵심 원칙:**
- 기능이 아닌 동기에 집중
- "무엇을 만들까"보다 "왜 사용하는가"에 답함
- 경쟁사는 같은 Job을 해결하는 모든 대안

---

## Job Statement 작성법

```
[동사] + [목적어] + [맥락/상황]
```

**예시:**
- ❌ 나쁜 Job: "사용자가 파일을 업로드하길 원한다"
- ✅ 좋은 Job: "팀 미팅 전에 최신 보고서를 동료들과 빠르게 공유하고 싶다"

**3가지 레이어:**
1. **기능적 Job**: 실제로 완수하려는 과업 ("보고서 공유")
2. **감정적 Job**: 느끼고 싶은 감정 ("전문적으로 보이고 싶다")
3. **사회적 Job**: 다른 사람에게 보이고 싶은 모습 ("유능한 팀원으로 인식되고 싶다")

---

## JTBD 인터뷰 방법

**핵심 질문:**
1. "마지막으로 [제품/서비스]를 사용했을 때를 떠올려보세요. 무엇을 하려고 했나요?"
2. "그 전에는 어떻게 이 일을 처리했나요?"
3. "이 방법을 선택한 이유는 무엇인가요?"
4. "무엇이 가장 불편했나요?"
5. "완벽한 솔루션은 어떤 모습일까요?"

**타임라인 매핑:**
```
[트리거 발생] → [대안 탐색] → [결정] → [사용] → [평가]
     ↓                ↓           ↓        ↓        ↓
  무엇이        어떤 옵션을    왜 선택    어떻게    결과에
  동기화?       고려했나?      했나?      사용?    만족?
```

## JTBD 분석 템플릿

```
Job Statement: [완수하려는 일]

트리거 상황:
- 언제 이 Job이 발생하는가?
- 어떤 상황이 행동을 유발하는가?

현재 해결책:
- 사용자가 현재 이 Job을 어떻게 처리하는가?
- 어떤 대안/경쟁사를 사용하는가?

불만족 요소:
- 현재 해결책의 어떤 점이 부족한가?
- 어떤 부분에서 마찰이 발생하는가?

성공 기준:
- 사용자에게 "완료"의 의미는?
- 어떤 결과를 원하는가?

기회:
- 우리 제품이 더 잘 해결할 수 있는 부분은?
```

## JTBD vs 페르소나 비교

| 기준 | 페르소나 | JTBD |
|------|---------|------|
| 초점 | 사람의 특성 | 달성하려는 목표 |
| 질문 | "누가 우리 사용자인가?" | "사용자가 왜 우리 제품을 쓰는가?" |
| 활용 | 디자인, 마케팅 | 기능 설계, 포지셔닝 |
| 경쟁 범위 | 동일 카테고리 | 같은 Job을 해결하는 모든 대안 |

**함께 활용:**
페르소나로 "누가"를 정의하고, JTBD로 "왜"를 이해한다.

## 실전 예시

**유료 구독 결정 맥락 — Spotify**

```
Job: "운동하는 동안 에너지를 유지하기 위해 방해 없이 음악을 듣고 싶다"

트리거: 헬스장 가기 전
현재 해결책: 광고 있는 무료 Spotify / YouTube
불만족: 운동 중 광고로 집중력 분산

성공 기준: 광고 없이 오프라인에서도 재생
우리의 기회: 프리미엄 구독 전환
```

## 활용 시점

✅ 적합:
- 새로운 기능/제품 아이디어 검증
- 경쟁사와의 차별화 포인트 발견
- 마케팅 메시지 개발
- 사용자 인터뷰 설계

❌ 주의:
- JTBD는 정량적 우선순위화 도구가 아님
- 기능 목록 작성보다 인사이트 도출에 적합
- RICE나 Kano와 함께 사용해 우선순위 결정
