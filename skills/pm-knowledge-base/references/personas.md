# 유저 페르소나 (User Personas)

## 개요

유저 페르소나는 실제 사용자 리서치를 기반으로 만든 가상의 사용자 대표 프로필이다.
제품 결정 시 "이 사용자에게 맞는가?"라는 질문의 기준이 된다.

**목적:**
- 팀 전체가 동일한 사용자를 상상하며 작업
- 기능 우선순위 결정 시 사용자 관점 유지
- 공감 기반 설계 지원

---

## 페르소나 템플릿

```
이름: [가상 이름]
나이: [나이대]
직업: [직책/역할]
거주지: [지역]

배경:
[2-3줄 소개 — 삶과 맥락]

목표:
- 주요 목표 1
- 주요 목표 2
- 주요 목표 3

불편함 (Pain Points):
- 현재 겪는 문제 1
- 현재 겪는 문제 2
- 현재 겪는 문제 3

행동 패턴:
- 기술 사용 습관
- 구매 결정 방식
- 정보 탐색 방법

선호 채널:
- 사용하는 기기/앱/플랫폼

인용구:
"이 페르소나가 직접 말할 법한 문장"
```

---

## 좋은 페르소나 vs 나쁜 페르소나

| 기준 | 나쁜 페르소나 | 좋은 페르소나 |
|------|-------------|-------------|
| 근거 | 팀의 가정 기반 | 실제 인터뷰/데이터 기반 |
| 구체성 | "30대 직장인" | "서울 강남 거주 35세 마케팅 팀장" |
| 목표 | "편리함 원함" | "주간 보고서 작성 시간 50% 단축 원함" |
| Pain Point | "불편하다" | "팀별 데이터 취합에 매주 3시간 소요" |

---

## 페르소나 작성 방법

**1단계 — 데이터 수집:**
- 사용자 인터뷰 (5-10명)
- 설문 조사
- 고객 지원 티켓 분석
- 행동 데이터 (GA, Mixpanel 등)

**2단계 — 패턴 식별:**
- 공통 목표 그룹화
- 주요 Pain Point 클러스터링
- 행동 패턴 구분

**3단계 — 페르소나 작성:**
- 각 주요 사용자 세그먼트당 1개
- 보통 2-4개로 제한 (너무 많으면 집중력 분산)

**4단계 — 우선순위 페르소나 지정:**
- 주요 페르소나 (Primary): 제품이 가장 먼저 만족해야 할 대상
- 보조 페르소나 (Secondary): 추가로 고려할 대상

---

## 예시: SaaS 프로젝트 관리 도구

```
이름: 이지수
나이: 34세
직업: IT 스타트업 제품팀장
거주지: 서울 마포구

배경:
3년차 PM으로 5명 팀을 이끌고 있다. 주간 스프린트를 운영하며
Jira, Slack, Google Meet를 매일 사용한다. 월 2회 임원진에게
제품 진행 상황을 보고한다.

목표:
- 팀 생산성 향상
- 경영진에게 명확한 진행 상황 보고
- 출시 일정 준수

불편함:
- 여러 툴에 흩어진 정보를 매번 취합해야 함
- 주간 보고서 작성에 2-3시간 소요
- 팀원 작업 현황을 실시간으로 파악하기 어려움

인용구:
"회의 준비보다 실제 제품 작업에 더 많은 시간을 쓰고 싶어요"
```

---

## 활용 시점

✅ 적합:
- 신제품/기능 기획 시 사용자 대상 정의
- 기능 우선순위 논쟁 해결 ("지수는 이 기능이 필요할까?")
- 사용자 스토리 작성 시 페르소나 기반 역할 정의
- 디자인 리뷰 시 사용자 관점 확인

❌ 부적합:
- 데이터 없이 직관만으로 만든 페르소나는 오히려 오해 유발
- 너무 많은 페르소나는 팀 집중력 분산
